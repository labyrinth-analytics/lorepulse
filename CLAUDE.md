# LorePulse (v0.1.0) - ALPHA

dbt pipeline health metrics MCP server. Surfaces test pass rates and source freshness
from local dbt artifacts. Part of the Lore product family by Labyrinth Analytics Consulting.

## What This Product Does

LorePulse reads dbt `target/` artifacts after a dbt run and exposes pipeline health
metrics via MCP tools. Data engineers can ask Claude "are my tests passing?" or
"which sources are stale?" without leaving their AI session.

Phase 1 (this version): local artifact reads only. No dbt Cloud API, no SQLite,
no credentials -- LorePulse never touches the warehouse directly.

Phase 2 (future): dbt Cloud API integration, run history tracking (SQLite),
schema drift detection.

## Architecture

- Stack: FastMCP, stdlib only (json, pathlib)
- Transport: stdio
- Data source: local dbt target/ artifacts (run_results.json, sources.json)
- No database -- reads fresh artifacts on every call
- MCP tools: 3

## Key Files

- `lorepulse/server.py` -- MCP server entry point (3 `@mcp.tool()` decorators)
- `lorepulse/__init__.py` -- package init, version
- `pyproject.toml` -- package config, entry point: `lorepulse.server:main`
- `requirements.txt` -- pinned: mcp[cli]==1.26.0

## MCP Tools

| Tool | Description | Input | Key output fields |
|------|-------------|-------|-------------------|
| `get_test_results` | Test pass/fail from target/run_results.json | project_path (optional) | total, passed, failed, errored, warned, failures[] |
| `get_model_freshness` | Source freshness from target/sources.json | project_path (optional) | total, fresh, stale, errored, sources[] |
| `get_pipeline_summary` | Rolled-up health dashboard | project_path (optional) | health, status, tests{}, freshness{}, failures[], stale_sources[] |

## Design Decisions

- `project_path` defaults to `os.getcwd()` -- agent passes the dbt project directory
  if it differs from the current working directory
- No subprocess calls to dbt -- reads artifacts directly. Avoids environment/credential
  issues at the MCP layer; user runs dbt themselves (or via CI) before querying LorePulse.
- test node filter: `unique_id.startswith("test.")` -- catches all dbt generic + singular tests
- freshness "stale" = status of "warn" or "error" (not just "error" -- warn means late, which
  is already a problem for most data teams)
- `get_pipeline_summary` calls the other two tools internally; no duplicate parsing code

## Known Issues

- Phase 1 only reads local artifacts. dbt Cloud users must download run results manually
  or wait for v0.2.0 dbt Cloud API integration.
- sources.json is only generated by `dbt source freshness` -- not by `dbt run` or `dbt test`.
  If the file is missing, get_model_freshness returns an informative error.
- run_results.json contains ALL node types from the last dbt invocation. If the user ran
  `dbt run` (not `dbt test`), the file will exist but have zero test nodes -- this returns
  total=0, which is correct but may confuse users. The last_run timestamp disambiguates.
